# Analysis: PR #2803 тАФ TRTLLM Fused MoE Routing Refactor

**Branch under review:** `refactor_routing` (commit `97b6f0e7`)
**Issue:** [#2732](https://github.com/flashinfer-ai/flashinfer/issues/2732)
**Minimal-fix branch:** `minimal-fix-2732` (based on `HEAD~1` of main)

---

## Summary

PR #2803 is a ~3,268-line architectural refactor of the TRTLLM fused MoE routing layer.
The bulk is mechanical renaming/reorganisation; the substantive fixes address **four bugs**
(AтАУD) identified in issue #2732.

This document separates the mechanical changes from the behaviour-changing ones and
describes the **minimal fix** implemented on branch `minimal-fix-2732`.

---

## Routing Method Taxonomy

| Enum value | Name | Pre-processing | Post-processing |
|---|---|---|---|
| 0 | `Default` | Softmax тЖТ TopK | none |
| 1 | `DeepSeekV3` | sigmoid + bias + group-TopK | none |
| 2 | `Llama4` | Top1 + sigmoid | none |
| 3 | `Renormalize` | TopK тЖТ Softmax | none |
| 4 | `RenormalizeNaive` | Softmax тЖТ TopK | divide by sum |
| 5 | `TopK` | TopK only | none |
| **6** | **`SigmoidRenorm`** | **Sigmoid тЖТ TopK** | **divide by sum** |
| 7 | `Unspecified` | тАФ | тАФ |

`SigmoidRenorm` was the **missing routing path** that triggered issue #2732.

## Four Bugs Fixed

### Bug A тАФ `SigmoidRenorm` not dispatched to `routingRenormalize`

**File:** `csrc/trtllm_fused_moe_runner.cu`

**Before:**
```cpp
} else if (routingMethodType == RoutingMethodType::Default
           || routingMethodType == RoutingMethodType::Renormalize
           || routingMethodType == RoutingMethodType::RenormalizeNaive
           || routingMethodType == RoutingMethodType::TopK) {
```

`SigmoidRenorm` fell through to an unhandled branch and was silently skipped.

**After:** The condition includes `RoutingMethodType::SigmoidRenorm`.

---

### Bug B тАФ Routing logits dtype hardcoded to `Fp32`

**File:** `csrc/trtllm_fused_moe_runner.cu`

**Before:**
```cpp
routingData.mDtypeScore = btg::Dtype::Fp32;           // always fp32
routingData.mPtrScores = reinterpret_cast<float*>(routingLogits);
```

The kernel always assumed float32 logits regardless of the actual tensor dtype.

**After:**
```cpp
routingData.mDtypeScore = dtypeLogits;   // passed from launcher
routingData.mPtrScores = routingLogits;  // void* forwarded directly
```

The launcher (`FusedMoeLauncher::prepare_routing`) now inspects `routing_logits.dtype()`
and sets `mRoutingLogitsDtype` to `Bfloat16` or `Fp32` accordingly.

---

### Bug C тАФ `norm_topk_prob` not exposed to Python callers

**Files:** `csrc/trtllm_fused_moe_kernel_launcher.cu`, `flashinfer/fused_moe/core.py`

**Before:** `norm_topk_prob` was hardcoded to `true` in the C++ runner; Python had no way
to override it.

**After:**
- `bool norm_topk_prob = true` added to all five `init()` overloads and forwarded via
  `Runner::run()` down to `routingRenormalize::Data::mNormTopkProb`.
- All five Python API functions (`trtllm_bf16_moe`, `trtllm_fp8_per_tensor_scale_moe`,
  `trtllm_fp8_block_scale_moe`, `trtllm_fp4_block_scale_moe`,
  `trtllm_mxint4_block_scale_moe`) accept `norm_topk_prob: bool = True`.

---

### Bug D тАФ `SigmoidRenorm` pre-processing not implemented in the kernel

**File:** `csrc/trtllm_fused_moe_routing_renormalize.cu`

The `routingTopKExperts` function had a `DoSoftmaxBeforeTopK` compile-time path for
Softmax + SumNorm, but no way to select Sigmoid instead of Softmax.

**After:** A runtime `bool doSigmoidBeforeTopK = false` parameter was added.
Inside the `DoSoftmaxBeforeTopK` branch:
```cpp
if (doSigmoidBeforeTopK) {
    for (int i = 0; i < VecSize; i++)
        if (idx[i] < numExperts)
            score[i] = DataType{sigmoid_accurate(float{score[i]})};
} else {
    calcSoftmax(warp, score);
}
```
The existing sum-normalise post-step (gated by `DoSoftmaxBeforeTopK`) is reused for free.

## What PR #2803 Does Differently (Mechanical Refactor)

Beyond the four bug fixes, PR #2803 also:

- Renames internal structs/methods for consistency
- Consolidates duplicated routing-config logic into a single helper
- Extracts `RoutingConfig` as a first-class struct passed through the call stack
- Adds additional routing paths and kernel specialisations that go beyond the minimal fix

These mechanical changes are **not needed** to fix issue #2732 and were omitted from the
minimal fix to keep the diff reviewable and risk-minimal.

## Key Kernel Data-Flow for `SigmoidRenorm`

```
Python: trtllm_bf16_moe(..., routing_method_type=RoutingMethodType.SigmoidRenorm,
                              norm_topk_prob=True)
  тФФтФА FusedMoeLauncher::init(norm_topk_prob=true)
       тФФтФА prepare_routing() тЖТ mRoutingLogitsDtype = Bfloat16 (from tensor.dtype())
            тФФтФА Runner::run(dtypeLogits=Bfloat16, normTopkProb=true)
                 тФФтФА routingRenormalize::Data {
                       mDtypeScore    = Bfloat16,    // Bug B fix
                       mDoSoftmaxBeforeTopK = true,  // gates SumNorm post-step
                       mDoSigmoidBeforeTopK = true,  // Bug D fix
                       mNormTopkProb  = true,        // Bug C fix
                    }
                    тЖТ routingTopKExperts<DoSoftmaxBeforeTopK=true>(
                          ..., doSigmoidBeforeTopK=true)
                      тФЬтФА read logits as bf16
                      тФЬтФА sigmoid_accurate() per score   (instead of calcSoftmax)
                      тФЬтФА select top-K
                      тФФтФА divide each score by sum of selected scores
```
